## Core premise

AI has lowered the barrier to building software, so engineering time is no longer the scarce resource — **taste and judgment** are. Career boundaries between product, design, and engineering are blurring; the strongest builders can speak to *why* something is built, whether the design is right, and how to build it well.

## Overview

The course moves through the full loop of applied product thinking:

- **Critique** — pick a real flow, break down the UX (information architecture, friction, visual design)
- **Hypothesize** — frame a specific problem and a proposed change, grounded in user needs
- **Test** — design an A/B test: metric, variants, sample size, success criteria
- **Learn** — document outcomes and what they imply for the next iteration



## Session 1



### Teaching goals

- Read a design critically — identify what it's trying to accomplish and how well it does that
- Talk about it confidently, not just "I like it" / "I don't like it"
- Propose and validate improvements with reasoning



### Key concept — outcomes

Every design exists to serve an **outcome** at the intersection of what the user is trying to accomplish and what the business needs. Measure outcomes across three time horizons:


| Horizon         | Question                           |
| --------------- | ---------------------------------- |
| **Near-term**   | Did the user get through the task? |
| **Medium-term** | Did they come back?                |
| **Long-term**   | Do they still find value?          |


The deck uses a grocery-price-increase example: a near-term win (revenue spike) can mask a long-term loss (customers quietly leaving).

### Workshop exercises

Students examine real screenshots and discuss what success looks like for the user and the business at each time horizon, then compare answers in small groups:

- Facebook Marketplace listing flow
- UberEats checkout
- LinkedIn notifications
